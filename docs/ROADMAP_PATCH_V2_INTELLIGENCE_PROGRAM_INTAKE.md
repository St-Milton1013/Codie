## Program map

### Program A: Relationship Intelligence

Relationship Intelligence remains its own Class 2 measured-evidence program.
It may calculate reproducible typed relationship metrics from canonical
population manifests. It may not generate recommendations, causal claims,
package truth, rules truth, or strategic conclusions.

Future contracts should progress through these stages:

1. Boundary, ontology, metric, and population contract.
2. Schema and repository migration contract.
3. Pure metric packet and deterministic math implementation.
4. Canonical population and repository integration.
5. Evidence Fusion projection.
6. Reporting and checkpoint validation.

The following remain deferred unless separately accepted:

- automated package discovery;
- graph embeddings;
- causal inference;
- replacement prediction;
- a dedicated graph database;
- cloud synchronization;
- public sharing.

Tournament Exposure remains a separate core V2 measured-evidence program. It
may consume approved population records but must not be hidden inside
Relationship Intelligence.

### Program B: Jin, Theory, Rules, and Corrections

This program coordinates dependencies without collapsing their trust
boundaries.

Recommended contract order:

1. Cross-specification boundary and decision contract.
2. Fixed Jin regression corpus schema and deterministic evaluation contract.
3. Rules authority, legality, and bounded interaction contract.
4. Local-first model profile, redaction, consent, and routing contract.
5. Minimal User Correction Ledger core contract.
6. Theory source registry, rights, immutable source version, and citation
   contract.
7. Reviewed claim, typed graph, contradiction, translation, and retrieval
   contract.
8. Jin intent, scope, query-plan, evidence-gate, and legality-gate contract.
9. Jin writer, auditor, deterministic finalizer, and answer-packet contract.
10. Experiment and permitted user-context write contract.
11. Judge-training and curriculum contract.
12. Program checkpoint and release acceptance.

Required dependency rules:

- The fixed regression corpus structure must exist before model-backed Jin
  output is accepted.
- The Rules Layer must exist before Jin claims rules validation or simulator
  trace validation.
- Model invocation must use an accepted model profile and deny cloud by
  default.
- The Correction Ledger must preserve authority ceilings and narrowest-scope
  application before Jin can apply corrections.
- Theory retrieval must use approved, attributed, rights-cleared, versioned
  claims before Jin can present an author's framework.
- Jin may consume Relationship Intelligence and Tournament Exposure only as
  labeled evidence references.
- Jin may not write canonical evidence, measurements, confidence tables,
  legality truth, or persisted recommendations.

### Program C: Presentation and export

The Decision Evidence Panel and Jin Chat Experience are complementary
projections of shared packet contracts:

- the Decision Evidence Panel presents conclusions and their evidence;
- Jin Chat presents governed conversation, comparison, challenge, and
  experiment workflows.

Neither surface may calculate evidence, resolve corrections, strengthen
confidence, or create persisted recommendations.

The Knowledge Vault remains an exported, non-canonical projection. File writers,
incremental filesystem mutation, ZIP creation, Obsidian plugin behavior, and
import-back behavior require later explicit contracts.

Recommended order:

1. Shared read models and view-model boundary.
2. Desktop Deck and Analysis workspaces.
3. Decision Evidence Panel.
4. Jin conversation and evidence inspection.
5. Staged experiment and correction workflows.
6. Knowledge Vault planner and renderer.
7. Separately contracted safe file writer.
8. Accessibility, privacy, and adversarial checkpoint.

Mobile execution remains deferred. A future mobile surface is read-only unless
separately contracted.

## Cross-program invariants

Every future contract must preserve:

- official rules and canonical truth outrank corrections and theory;
- canonical records precede measured evidence;
- measured evidence precedes persisted conclusions;
- user decks do not enter global population evidence without explicit
  authorization;
- theory explains but does not replace evidence;
- authors are provenance subjects, not truth authorities;
- corrections apply at the narrowest valid scope;
- unknown, unsupported, stale, and conflicting states remain visible;
- no LLM may mutate canonical, measured, legality, correction, or
  recommendation records;
- cloud processing is deny-by-default and never required;
- private or licensed source text is not transmitted or exported without an
  accepted rights policy;
- persisted outputs retain versioned inputs, provenance, and replay identity;
- tests do not require live network access or paid API keys.

## Consolidated unresolved decisions

The proposals contain decisions that must be resolved in bounded contracts,
not during implementation.

### Relationship Intelligence

- default commander population and placement scope;
- default stability and low-sample thresholds;
- weighted metrics in the first release;
- storage of pair-detail provenance;
- core versus follow-on metric families;
- graph storage versus pair-detail-only storage;
- theory links in the first migration.

### Rules and legality

- official Comprehensive Rules acquisition location and format;
- date-aware Commander legality authority;
- initial release-note and tournament-policy coverage;
- refresh cadence and historical retention;
- first supported continuous-effect capability manifest;
- persistence policy for rules answers and lesson submissions;
- blocking behavior when authority versions change.

### Model profiles

- supported local inference runtimes and minimum hardware profiles;
- default local models and quantization tiers;
- whether any cloud providers are admitted;
- provider retention verification and request retention;
- local and cloud audit thresholds;
- private-data consent duration;
- benchmarking as a release gate.

### Correction Ledger

- core migration and repository boundary;
- activation approval for correction candidates;
- review requirements for authority-sensitive corrections;
- revalidation fan-out and conflict resolution policy;
- retention, redaction, and multi-user isolation.

### Theory Corpus

- graph and source-asset storage;
- private embeddings;
- acceptable transcription tools;
- review quorum and promotion thresholds;
- router weights and primary-source thresholds;
- community snapshot retention;
- quotation limits by rights class;
- curriculum storage;
- graph release and patch policy;
- initial tempo source set.

### Jin

- default local model profiles;
- mandatory theory and audit thresholds;
- answer and conversation retention;
- correction and experiment confirmation;
- localhost-only API policy;
- deterministic no-model capability;
- minimum theory retrieval required for the first complete release.
